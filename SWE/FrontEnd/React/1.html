React Hooks
Mastery Guide

A complete technical reference for React's core hooks — covering state, side effects, refs, memoization, and async patterns with real-world examples.

#useState #useRef #useEffect #useMemo #useCallback #async patterns

React Hooks, introduced in v16.8, let you use state and other React features inside function components — replacing class components for most use cases. Each hook solves a specific problem: managing local state, synchronizing with external systems, caching expensive computations, or stabilizing function references. Knowing when to reach for each one — and when to skip it — is what separates competent React from expert React.

01 ——
useState
Persist and update values that trigger re-renders when changed
BEGINNER
What it is
A hook that returns a stateful value and a setter function. React re-renders the component whenever the setter is called with a new value.
Why use it
Any value that should be displayed in the UI and can change over time — form inputs, toggles, counters, selected items, loading flags.
Real-world use case
A search bar that updates results as the user types, controlled form fields, a modal open/close toggle, pagination current page.

useState — patterns & pitfalls 
import { useState } from 'react';

// ① Basic usage
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(c => c + 1)}>
      Clicked {count} times
    </button>
  );
}

// ② Functional update form (safe for async/batched updates)
setCount(prev => prev + 1); // ✅ always based on latest state
setCount(count + 1);         // ⚠️  stale closure risk in async code

// ③ Lazy initializer — runs once, not every render
const [state, setState] = useState(() => expensiveComputation());

// ④ Object state — always spread to avoid mutations
const [user, setUser] = useState({ name: '', age: 0 });
setUser(prev => ({ ...prev, name: 'Alice' })); // ✅ immutable update

// ⑤ Batching — React 18 batches all updates automatically
setA(1); setB(2); // one re-render, not two
✅Performance: Use the functional updater form setState(prev => ...) when new state depends on old state — prevents bugs from stale closures in async callbacks.
⚠️Mistake: Mutating state directly (state.count++) won't trigger a re-render. React uses reference equality — always return a new object or array.
💡Pro tip: Derive values from state rather than duplicating state. If you have items and filteredItems, compute filteredItems with useMemo instead of syncing two state vars.
⏭️When to skip: If the value doesn't need to trigger a re-render (e.g., a timer ID, a DOM ref, a previous value tracker), use useRef instead.
02 ——
useRef
A mutable container that persists across renders without causing them
BEGINNER
What it is
Returns a plain object {current: value} that persists for the full component lifetime. Mutating .current never triggers a re-render.
Why use it
Two jobs: (1) hold a reference to a DOM node for imperative access, (2) store any mutable value that shouldn't cause re-renders (timers, previous values, flags).
Real-world use case
Auto-focus an input, measure element dimensions, store a setInterval ID for cleanup, track previous prop value, or accumulate values without re-render churn.

useRef — DOM access & mutable values 
import { useRef, useEffect } from 'react';

// ① DOM node access
function AutoFocusInput() {
  const inputRef = useRef(null);

  useEffect(() => {
    inputRef.current?.focus(); // safe optional chaining
  }, []);

  return <input ref={inputRef} placeholder="Auto-focused" />;
}

// ② Store mutable value — timer ID
function Stopwatch() {
  const [elapsed, setElapsed] = useState(0);
  const timerRef = useRef(null);

  const start = () => {
    timerRef.current = setInterval(() =>
      setElapsed(e => e + 1), 1000
    );
  };
  const stop = () => clearInterval(timerRef.current);

  return <><button onClick={start}>Start</button><button onClick={stop}>Stop</button></>;
}

// ③ Track previous prop value
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => { ref.current = value; }); // runs after render
  return ref.current; // returns previous render's value
}
✅Performance: Use useRef for values that change frequently but don't need to drive the UI (e.g., scroll position, mouse coords during drag) to avoid unnecessary re-renders.
⚠️Mistake: Reading ref.current during render is unreliable — refs are set after painting. Only read refs in event handlers and effects, never in the render phase.
💡Pro tip: Use useRef to "escape" stale closures in event listeners: store the latest callback in a ref and call ref.current() from the listener instead.
⏭️When to skip: If changing the value should update the UI, use useState. useRef is for side-channel data that the render output doesn't depend on.
03 ——
useEffect
Synchronize a component with an external system after rendering
INTERMEDIATE
What it is
Runs a side effect after the browser has painted. Accepts a setup function and a dependency array. Returns an optional cleanup function called before the next run or unmount.
Why use it
Data fetching, subscriptions, DOM manipulation, analytics events, syncing with localStorage, setting up/tearing down WebSocket connections, third-party library initialization.
Real-world use case
Fetch user data when a userId changes, subscribe to a Firestore document, update document.title when a route changes, register and deregister event listeners.

useEffect — dependencies & cleanup 
import { useEffect, useState } from 'react';

// ① Three dependency array forms
useEffect(() => { /* runs after every render */ });
useEffect(() => { /* runs once on mount */ }, []);
useEffect(() => { /* runs when userId changes */ }, [userId]);

// ② Data fetch with cleanup (abort controller)
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    const controller = new AbortController();

    fetch(`/api/users/${userId}`, { signal: controller.signal })
      .then(r => r.json())
      .then(setUser)
      .catch(err => {
        if (err.name !== 'AbortError') console.error(err);
      });

    return () => controller.abort(); // cleanup cancels in-flight request
  }, [userId]);

  return user ? <div>{user.name}</div> : <p>Loading...</p>;
}

// ③ Event listener with cleanup
useEffect(() => {
  const handler = (e) => setKey(e.key);
  window.addEventListener('keydown', handler);
  return () => window.removeEventListener('keydown', handler);
}, []);
✅Performance: Always return a cleanup function for subscriptions, timers, and event listeners to prevent memory leaks. StrictMode will double-invoke effects in dev to expose missing cleanups.
⚠️Mistake: Setting state unconditionally inside an effect that lists that state as a dependency creates an infinite loop. Add a condition or restructure the logic.
💡Pro tip: Consider useLayoutEffect if your effect reads layout (DOM measurements) and writes synchronously — it fires before paint, preventing flicker.
⏭️When to skip: Don't use useEffect for transforming data for render (use useMemo) or handling user events (use event handlers). Effects are for synchronization, not computation.
04 ——
useMemo
Cache an expensive computed value between renders
INTERMEDIATE
What it is
Memoizes the return value of a function. Re-executes the function only when dependencies change. Returns the cached value on all other renders.
Why use it
Skip expensive recalculations (sorting, filtering, math), maintain referential equality for arrays/objects passed to memoized child components or used in effect deps.
Real-world use case
Filtering/sorting a 10,000-row table, computing derived analytics from raw data, stabilizing an options array passed to a React.memo child to prevent re-renders.

useMemo — memoizing expensive work 
import { useMemo, useState } from 'react';

// ① Expensive computation
function DataTable({ rows, searchTerm, sortKey }) {
  const processedRows = useMemo(() => {
    const filtered = rows.filter(r =>
      r.name.toLowerCase().includes(searchTerm.toLowerCase())
    );
    return [...filtered].sort((a, b) => a[sortKey] > b[sortKey] ? 1 : -1);
  }, [rows, searchTerm, sortKey]); // only recomputes when these change

  return processedRows.map(row => <Row key={row.id} data={row} />);
}

// ② Stabilizing reference equality for React.memo children
const Child = React.memo(({ options }) => <Select options={options} />);

function Parent({ theme }) {
  // Without useMemo, new array reference on every render → Child re-renders
  const options = useMemo(() => [
    { value: 'a', label: 'Option A' },
    { value: 'b', label: 'Option B' },
  ], []); // empty deps — never recreated

  return <Child options={options} />;
}

// ③ When NOT to useMemo (premature optimization)
const x = useMemo(() => a + b, [a, b]); // ❌ trivial math, just write: a + b
✅Performance: Profile before memoizing. useMemo has overhead (comparison, memory). Only memoize when a React Profiler measurement shows the computation is genuinely expensive (>1ms).
⚠️Mistake: Forgetting a dependency causes the cached value to go stale. The ESLint react-hooks/exhaustive-deps rule catches this — enable it in all React projects.
💡Pro tip: useMemo is also useful for stabilizing object/array deps in useEffect. If you construct an object inside a component and use it in an effect, memoize it first.
⏭️When to skip: Primitive values (strings, numbers, booleans) are compared by value — no memoization needed. Skip for computations under ~0.1ms, that's cheaper than the comparison.
05 ——
useCallback
Memoize a function reference to preserve identity across renders
INTERMEDIATE
What it is
Returns a memoized version of a callback. The function is recreated only when dependencies change. Essentially useMemo(() => fn, deps) where the cached value is a function.
Why use it
Prevent unnecessary re-renders of memoized child components that receive callbacks as props. Stabilize function references used in useEffect dependency arrays.
Real-world use case
An onChange handler passed to a heavy React.memo list item, a debounced search callback, a click handler that's a dep in a useEffect subscription.
Parent renders
→
useCallback(fn, deps)
→
Same ref if deps unchanged
→
Child skips re-render
useCallback — stable references 
import { useCallback, useState, memo } from 'react';

// ① Heavy child that should only re-render when its data changes
const ListItem = memo(({ item, onDelete }) => (
  <div>{item.name} <button onClick={() => onDelete(item.id)}>✕</button></div>
));

function TodoList() {
  const [todos, setTodos] = useState([]);

  // Without useCallback: new function ref each render → memo is useless
  const handleDelete = useCallback((id) => {
    setTodos(prev => prev.filter(t => t.id !== id));
  }, []); // empty deps — setTodos is stable from React

  return todos.map(t =>
    <ListItem key={t.id} item={t} onDelete={handleDelete} />
  );
}

// ② Stable dep in useEffect
const fetchData = useCallback(() => {
  return fetch(`/api/data?q=${query}`).then(r => r.json());
}, [query]); // new function only when query changes

useEffect(() => { fetchData().then(setData); }, [fetchData]);
✅Performance: useCallback is only effective when the receiving component is wrapped in React.memo. Without that, the child re-renders regardless of prop reference stability.
⚠️Mistake: Wrapping every function in useCallback "just in case" adds overhead with no benefit. This is one of the most common React performance anti-patterns.
💡Pro tip: Use the "stable setter" pattern: since React guarantees state setters are stable references, you can often use [] as deps if your callback only calls setters via functional update form.
⏭️When to skip: Skip if the child isn't memoized, if the callback isn't a useEffect dep, or if the parent only renders rarely. Most inline handlers don't need this.

Async Patterns
Handle async operations, race conditions, and data fetching correctly
ADVANCED
What it is
React doesn't have a native useAsync hook. Instead, you compose useEffect, useState, and AbortController — or use libraries like React Query / SWR for production-grade async.
Why use it
Async operations in components need explicit loading, error, and data states, plus cleanup for race conditions (stale responses from previous renders arriving late).
Real-world use case
Search-as-you-type (racing fetch calls), paginated list loading with cancellation, form submission with optimistic UI, polling with exponential backoff.

Async patterns — race-safe data fetching & custom hook 
import { useState, useEffect, useCallback } from 'react';

// ① Custom useAsync hook — reusable pattern
function useAsync(asyncFn, deps) {
  const [state, setState] = useState({
    data: null, loading: true, error: null
  });

  useEffect(() => {
    let cancelled = false;
    setState(s => ({ ...s, loading: true, error: null }));

    asyncFn()
      .then(data => { if (!cancelled) setState({ data, loading: false, error: null }); })
      .catch(err  => { if (!cancelled) setState({ data: null, loading: false, error: err }); });

    return () => { cancelled = true; }; // prevents stale state on unmount
  }, deps); // eslint-disable-line react-hooks/exhaustive-deps

  return state;
}

// ② Usage — clean, declarative
function UserCard({ userId }) {
  const fetchUser = useCallback(
    () => fetch(`/api/users/${userId}`).then(r => r.json()),
    [userId]
  );
  const { data, loading, error } = useAsync(fetchUser, [fetchUser]);

  if (loading) return <Spinner />;
  if (error)   return <Error message={error.message} />;
  return <Profile user={data} />;
}

// ③ React 18+: use() hook for Suspense-native data fetching
// const data = use(fetchUserPromise);  ← throws promise, caught by <Suspense>
✅Performance: Consider React Query or SWR for production data fetching. They handle caching, deduplication, background refetch, and stale-while-revalidate out of the box.
⚠️Mistake: Making useEffect's callback async directly (useEffect(async () => ...)) causes the cleanup return value to be a Promise, not a function — React ignores it, causing leaks.
💡Pro tip: For search-as-you-type, debounce the query with a useEffect + setTimeout, and use AbortController to cancel in-flight requests when the query changes before they resolve.
⏭️When to skip: If you're using Next.js App Router or Remix, server components and loaders handle data fetching without useEffect. Client-side hooks are for genuinely client-only data.
