React Hooks
Deep Dive

Master the six hooks that will transform how you think about state, performance, and side effects in modern React.

useState useMemo useRef useCallback useEffect useAsync

Why Hooks Exist

Before Hooks (React 16.8, Feb 2019), sharing stateful logic between components required awkward patterns — higher-order components, render props — that created "wrapper hell". Hooks let you extract and reuse stateful logic without changing component hierarchy. They make code more readable, testable, and composable.

🔁
Reuse Logic
Extract stateful logic into custom hooks, share across components without HOC nesting.
🧩
Collocate Code
Group related code by concern, not by lifecycle method. useEffect replaced 3 lifecycle methods.
⚡
Performance
useMemo and useCallback give you surgical control over re-computation and reference stability.
📐
Only in Function Components
All rules apply: call at top level, only in React functions — never in loops, conditions, or nested functions.
01 /
useState
Local component state — the most fundamental hook
BUILT-IN
What it is

useState lets a function component "remember" a value between renders. It returns the current value and a setter function. Calling the setter triggers a re-render with the new value.

Why use it

Any time your UI needs to change in response to user interaction or data — form inputs, toggles, counters, modals — useState is the answer. Without it, function components are stateless.

Real-world use case

A shopping cart: const [cartItems, setCartItems] = useState([]) — you display cartItems and call setCartItems whenever a user adds or removes a product. Each change re-renders the cart component with fresh data.

Under the hood

React stores state in an internal "fiber" linked list. Each useState call corresponds to a hook slot by call order — hence the Rules of Hooks (never call conditionally). The setter function is referentially stable across renders.

Functional updates & batching

Pass a function setState(prev => prev + 1) when new state depends on previous — avoids stale closures. React 18 batches all setState calls automatically, even in async callbacks.

SYNTAX 
// Basic
const [state, setState] = useState(initialValue);

// With type annotation (TypeScript)
const [count, setCount] = useState<number>(0);

// Lazy initializer — function runs once
const [data, setData] = useState(() => JSON.parse(localStorage.getItem('data')));

// Functional update (safe when reading prev state)
setCount(prev => prev + 1);
EXAMPLE — Controlled Form + Toggle 
import { useState } from 'react';

function LoginForm() {
  const [email, setEmail]       = useState('');
  const [showPass, setShowPass] = useState(false);
  const [errors, setErrors]     = useState({});

  const handleSubmit = (e) => {
    e.preventDefault();
    if (!email.includes('@')) {
      // Functional update for object state
      setErrors(prev => ({ ...prev, email: 'Invalid email' }));
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={email}
        onChange={e => setEmail(e.target.value)}
      />
      <button type="button" onClick={() => setShowPass(p => !p)}>
        {showPass ? 'Hide' : 'Show'}
      </button>
      {errors.email && <span>{errors.email}</span>}
    </form>
  );
}
⚡ Performance Considerations
  • Every setState call schedules a re-render — batch related updates
  • Use lazy initializer for expensive initial computation
  • Split unrelated state into separate useState calls
⚠ Common Mistakes
  • Mutating state directly: state.items.push(x) — won't re-render
  • Stale closure: reading state in async without functional update
  • Setting state during render causes infinite loops
🚫 When NOT to Use
  • Cross-component shared state → use Context / Zustand / Redux
  • Server-side derived data that never changes locally
  • Non-UI values that don't cause re-renders → use useRef
💡 Pro Tip
  • Co-locate state as close to where it's used as possible
  • Consider useReducer when state logic grows complex
  • Objects & arrays need new references: spread to update

useMemo
Memoize expensive computations — skip redundant work
BUILT-IN
What it is

useMemo caches the result of a calculation. You provide a function and dependency array; React re-runs the function only when a dependency changes. Between renders, it returns the cached value.

Why use it

If a component re-renders often but a computation is expensive (sorting 10,000 items, building a complex tree), useMemo ensures you only do that work when the input data actually changes.

Real-world use case

A data table with client-side filtering: const filtered = useMemo(() => data.filter(row => row.name.includes(query)), [data, query]). The filter only re-runs when data or query changes, not on every parent re-render.

Referential stability

useMemo also prevents child components from re-rendering when you pass an object/array as props — the memoized value keeps the same reference. This complements React.memo on child components.

React Compiler (v19+)

The React Compiler (Forget) auto-memoizes components and values at compile time, potentially making manual useMemo obsolete. For now, profile before optimizing — premature memoization adds overhead.

⚡ Zero computation cost
SYNTAX 
const memoizedValue = useMemo(
  () => expensiveComputation(a, b),
  [a, b] // dependency array — recompute when a or b changes
);

// Stable object reference (prevents child re-renders)
const options = useMemo(() => ({ sort: 'asc', limit: 50 }), []);
// Empty deps → computed once, same reference forever
EXAMPLE — Filtered & Sorted List 
import { useState, useMemo } from 'react';

function ProductList({ products }) {
  const [query, setQuery]   = useState('');
  const [sortBy, setSortBy] = useState('price');

  // Only recalculates when products, query, or sortBy change
  const displayProducts = useMemo(() => {
    const filtered = products.filter(p =>
      p.name.toLowerCase().includes(query.toLowerCase())
    );
    return [...filtered].sort((a, b) =>
      a[sortBy] > b[sortBy] ? 1 : -1
    );
  }, [products, query, sortBy]);

  return (
    <div>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <p>Showing {displayProducts.length} of {products.length}</p>
      {displayProducts.map(p => <ProductCard key={p.id} product={p} />)}
    </div>
  );
}
⚡ Performance Considerations
  • useMemo itself has overhead — only use for genuinely expensive ops
  • Profile with React DevTools Profiler before adding useMemo
  • Pairs best with React.memo on child components
⚠ Common Mistakes
  • Missing deps: stale values silently bug your UI
  • Wrapping everything "just in case" — adds code noise
  • Using it for side effects — that's useEffect's job
🚫 When NOT to Use
  • Cheap computations: addition, string concat — overhead > benefit
  • When the component rarely re-renders anyway
  • To memo a value for side-effect purposes
💡 Pro Tip
  • Measure first — React is fast; most components don't need memo
  • useMemo returns a value; useCallback returns a function
  • React may discard the cache in the future — treat as an optimization hint

useRef
Mutable container that persists without triggering re-renders
BUILT-IN
What it is

useRef returns a plain object { current: value }. Unlike useState, mutating .current does NOT cause a re-render. The ref object persists for the full lifetime of the component.

Why use it

Two main uses: (1) Direct DOM access — focus an input, measure element size, integrate 3rd-party libraries. (2) Store mutable values that shouldn't trigger renders — timers, previous state snapshots, instance variables.

Real-world use case

Auto-focus a search input on mount, store an interval ID to clear it later, or access a canvas element for WebGL rendering. Libraries like react-hook-form use refs internally to avoid re-renders on every keystroke.

Callback refs

Pass a function as the ref prop: ref={node => { ... }}. Called with the DOM node when mounted, null when unmounted. Useful for measuring elements after paint (ResizeObserver pattern).

forwardRef & useImperativeHandle

Refs don't forward through components by default. Use forwardRef to expose a child's DOM node to a parent. useImperativeHandle lets you customize what the parent can access — expose only focus() instead of the entire input element.

SYNTAX 
// DOM ref
const inputRef = useRef(null);
<input ref={inputRef} />
inputRef.current.focus();  // direct DOM access

// Mutable instance variable (no re-render)
const timerRef = useRef(null);
timerRef.current = setInterval(tick, 1000);
clearInterval(timerRef.current);

// Track previous value
const prevCount = useRef(count);
useEffect(() => { prevCount.current = count; }, [count]);
EXAMPLE — Stopwatch with Interval Ref 
import { useState, useRef } from 'react';

function Stopwatch() {
  const [elapsed, setElapsed] = useState(0);
  const intervalRef = useRef(null);
  const startTime   = useRef(null);

  const start = () => {
    startTime.current = Date.now() - elapsed;
    intervalRef.current = setInterval(() => {
      setElapsed(Date.now() - startTime.current);
    }, 16); // ~60fps
  };

  const stop = () => clearInterval(intervalRef.current);

  const reset = () => {
    stop();
    setElapsed(0);
  };

  return (
    <div>
      <h2>{(elapsed / 1000).toFixed(2)}s</h2>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
      <button onClick={reset}>Reset</button>
    </div>
  );
}
⚡ Performance Considerations
  • Zero re-render cost for mutations — ideal for high-frequency updates
  • DOM refs let libraries like GSAP animate without React overhead
  • Use for animation state, scroll position, previous values
⚠ Common Mistakes
  • Reading ref.current during render — it may be null (DOM not yet attached)
  • Using ref when you need the UI to update — use useState instead
  • Forgetting refs don't trigger effects when they change
🚫 When NOT to Use
  • When you need the UI to reflect the value — use useState
  • For simple derived values — useMemo is cleaner
  • As an escape hatch to avoid proper state architecture
💡 Pro Tip
  • The ref object itself is stable — safe to include in dep arrays
  • Useful for storing latest callback: callbackRef.current = fn
  • Null-check before accessing: if (ref.current) ref.current.focus()

useCallback
Memoize function references — prevent unnecessary child re-renders
BUILT-IN
What it is

useCallback caches a function definition. On every render, JavaScript creates a new function object — even if the logic is identical. useCallback returns the same function reference as long as its dependencies haven't changed.

Why use it

When you pass a callback to a child component wrapped in React.memo, a new function reference on every render bypasses the memo. useCallback ensures the child only re-renders when the function's logic actually needs to change.

Real-world use case

A virtualized list with thousands of rows. Each row receives an onDelete callback. Without useCallback, every parent render creates a new function, forcing all rows to re-render. With it, rows only update when the deletion logic itself changes.

useCallback vs useMemo

useCallback(fn, deps) is syntactic sugar for useMemo(() => fn, deps). Use useCallback when your memoized value is a function — same caching mechanism, different intent signal.

The stale closure problem

A common pitfall: using a stale callback that closes over old state. Solution: use the functional updater form setState(prev => ...) inside callbacks, or include the state in the dependency array (accepting more re-memos).

Same fn reference

React.memo child
→
✓ Skips re-render
deps changed →
New fn reference
→

→

SYNTAX 
const memoizedFn = useCallback(
  () => { doSomething(a, b); },
  [a, b] // only recreate when a or b changes
);

// Equivalent (useMemo returning a function)
const sameThing = useMemo(
  () => () => { doSomething(a, b); },
  [a, b]
);
EXAMPLE — Optimized Virtualized List 
import { useState, useCallback, memo } from 'react';

// Memoized child — only re-renders if props change
const TodoItem = memo(({ todo, onDelete }) => (
  <li>
    {todo.text}
    <button onClick={() => onDelete(todo.id)}>✕</button>
  </li>
));

function TodoList() {
  const [todos, setTodos] = useState([...]);
  const [filter, setFilter] = useState('');

  // Stable reference — TodoItem won't re-render on filter change
  const handleDelete = useCallback((id) => {
    setTodos(prev => prev.filter(t => t.id !== id));
  }, []); // no deps — uses functional setState

  return (
    <ul>
      {todos
        .filter(t => t.text.includes(filter))
        .map(todo => (
          <TodoItem key={todo.id} todo={todo} onDelete={handleDelete} />
        ))}
    </ul>
  );
}
⚡ Performance Considerations
  • Only useful with React.memo — otherwise reference stability is meaningless
  • Each useCallback call has overhead; measure before adding
  • Works best on callbacks passed to many memoized children
⚠ Common Mistakes
  • Using stale state in callbacks — prefer functional updaters
  • Adding useCallback without React.memo on the child = wasted effort
  • Incorrect dependency array causes stale or re-created functions
🚫 When NOT to Use
  • Callbacks only used inside the component itself
  • When the child isn't wrapped in React.memo
  • Simple, lightweight components — overhead exceeds benefit
💡 Pro Tip
  • Always pair with React.memo — one without the other is wasteful
  • The eslint-plugin-react-hooks exhaustive-deps rule is your friend
  • useReducer's dispatch is already stable — no useCallback needed

useEffect
Synchronize with the outside world — side effects after render
BUILT-IN
What it is

useEffect lets you run code after React has rendered your component. It's the hook for "side effects" — things outside the React tree: fetching data, subscribing to events, setting up timers, manipulating the DOM, or syncing with external stores.

Why use it

React's render phase must be pure and free of side effects. useEffect is the designated escape hatch: it runs after the browser has painted, keeping renders fast while still letting you interact with the outside world safely.

Real-world use case

Fetching a user's profile on mount, subscribing to a WebSocket channel when a chat room opens (and unsubscribing when it closes), syncing a React state value to localStorage, or setting the document <title> dynamically based on the current route.

Strict Mode double-invoke

In React 18 Strict Mode (dev only), effects intentionally mount → unmount → remount to surface cleanup bugs. If your effect breaks, your cleanup function is missing or incorrect. This is intentional — fix the cleanup, not Strict Mode.

useLayoutEffect vs useEffect

useLayoutEffect fires synchronously after DOM mutations but before the browser paints — use it only for DOM measurements (getBoundingClientRect) or preventing visual flicker. For everything else, prefer useEffect to avoid blocking paint.

React 19: use() + Suspense

React 19 introduces the use() hook and async Server Components, shifting data fetching out of useEffect. For new projects, consider these patterns before reaching for data-fetching useEffect calls.

Dependency array deep-dive

The array is a list of reactive values the effect depends on. Omitting it means "run after every render". An empty array means "run once on mount". Missing deps cause stale closures; over-specifying deps causes infinite loops — the eslint exhaustive-deps rule catches both.

SYNTAX — All Variants 
// 1. Run after every render (no deps array)
useEffect(() => {
  doSomething();
});

// 2. Run once on mount, cleanup on unmount
useEffect(() => {
  const sub = subscribe(handler);
  return () => sub.unsubscribe(); // cleanup function
}, []); // empty array = mount only

// 3. Run when specific values change
useEffect(() => {
  fetchData(userId);
}, [userId]); // re-runs whenever userId changes

// 4. With AbortController for fetch cleanup
useEffect(() => {
  const controller = new AbortController();
  fetch(url, { signal: controller.signal });
  return () => controller.abort();
}, [url]);
EXAMPLE — Live Document Title + Event Listener 
import { useState, useEffect } from 'react';

function NotificationBell({ userId }) {
  const [unread, setUnread] = useState(0);
  const [online, setOnline] = useState(navigator.onLine);

  // Sync document title with unread count
  useEffect(() => {
    document.title = unread > 0
      ? `(${unread}) Notifications`
      : 'My App';
  }, [unread]);

  // Subscribe to online/offline events
  useEffect(() => {
    const goOnline  = () => setOnline(true);
    const goOffline = () => setOnline(false);
    window.addEventListener('online',  goOnline);
    window.addEventListener('offline', goOffline);
    return () => {  // ← cleanup: remove listeners on unmount
      window.removeEventListener('online',  goOnline);
      window.removeEventListener('offline', goOffline);
    };
  }, []); // [] = attach once, remove on unmount

  // Fetch notifications when userId changes
  useEffect(() => {
    if (!userId) return;
    const controller = new AbortController();

    fetch(`/api/notifications/${userId}`, { signal: controller.signal })
      .then(r => r.json())
      .then(data => setUnread(data.count))
      .catch(err => { if (err.name !== 'AbortError') console.error(err); });

    return () => controller.abort(); // cancel stale requests
  }, [userId]);

  return (
    <div>
      {!online && <Banner>You are offline</Banner>}
      <Bell count={unread} />
    </div>
  );
}
⚡ Performance Considerations
  • Effects run after paint — don't block the browser's rendering pipeline
  • Minimize effects; each one adds post-render overhead and complexity
  • Split independent concerns into separate useEffect calls for clarity and control
⚠ Common Mistakes
  • Missing cleanup: event listeners and subscriptions leak memory
  • Object/array deps: [{id}] creates a new ref every render → infinite loop
  • Setting state unconditionally in an effect with that state as a dep
🚫 When NOT to Use
  • Transforming data for rendering — compute during render or use useMemo
  • Handling user events — put logic directly in event handlers
  • Initializing something once — use lazy initializer in useState
💡 Pro Tip
  • Ask: "Could this be an event handler instead?" If yes, skip useEffect
  • Use AbortController for fetch calls — cleaner than isMounted flags
  • Each effect should do one thing — split them by concern, not lifecycle

useAsync
Custom hook — orchestrate async operations with full lifecycle state
CUSTOM HOOK
What it is

useAsync is a custom hook you build yourself (or install via libraries like react-use or ahooks). It wraps an async function and tracks its full lifecycle: idle → pending → success/error, exposing these as state values.

Why use it

Every API call in React needs the same boilerplate: loading state, error handling, data storage, cleanup on unmount, stale response prevention. useAsync centralizes this pattern so you write it once and reuse it everywhere.

Real-world use case

A user profile page: const { data, loading, error, execute } = useAsync(fetchUser). You get a consistent API for every data fetching scenario — search, forms, pagination — with race condition prevention built in.

Race condition prevention

When a user triggers multiple requests rapidly (search as you type), earlier responses may arrive after later ones. The pattern uses an isCancelled flag (or AbortController) — if the component unmounted or a new call started, the stale response is discarded.

Beyond this pattern

For production, consider TanStack Query (React Query) or SWR — they add caching, background refetching, deduplication, and optimistic updates. useAsync is the educational foundation; these libraries are the production evolution.

IMPLEMENTATION — Full useAsync Hook 
import { useState, useCallback, useRef, useEffect } from 'react';

function useAsync(asyncFn, immediate = false) {
  const [status, setStatus] = useState('idle'); // idle|pending|success|error
  const [data,   setData]   = useState(null);
  const [error,  setError]  = useState(null);
  const cancelledRef = useRef(false);

  useEffect(() => {
    cancelledRef.current = false;
    return () => { cancelledRef.current = true; }; // cleanup
  }, []);

  const execute = useCallback(async (...args) => {
    setStatus('pending');
    setData(null);
    setError(null);

    try {
      const result = await asyncFn(...args);
      if (!cancelledRef.current) {
        setData(result);
        setStatus('success');
      }
    } catch (err) {
      if (!cancelledRef.current) {
        setError(err);
        setStatus('error');
      }
    }
  }, [asyncFn]);

  useEffect(() => {
    if (immediate) execute();
  }, [immediate, execute]);

  return {
    execute,
    data,
    error,
    status,
    isIdle:    status === 'idle',
    isPending: status === 'pending',
    isSuccess: status === 'success',
    isError:   status === 'error',
  };
}
EXAMPLE — User Search with useAsync 
const fetchUser = (id) =>
  fetch(`/api/users/${id}`).then(r => r.json());

function UserProfile({ userId }) {
  const { execute, data: user, isPending, isError, error }
    = useAsync(fetchUser);

  useEffect(() => {
    execute(userId);
  }, [userId, execute]);

  if (isPending) return <Skeleton />;
  if (isError)   return <ErrorBanner message={error.message} />;
  if (!user)     return null;

  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
    </div>
  );
}
⚡ Performance Considerations
  • AbortController provides cleaner cleanup than boolean flags
  • Add caching logic to avoid redundant network requests
  • Consider React Suspense + use() for server-aligned data fetching
⚠ Common Mistakes
  • Not handling cleanup → setState after unmount (memory leak warning)
  • Race conditions: rapid calls overwriting each other's results
  • Not wrapping asyncFn in useCallback → execute re-created every render
🚫 When NOT to Use
  • Complex caching needs → use TanStack Query or SWR
  • Server components (Next.js App Router) — fetch directly in async components
  • One-off operations — inline useEffect + useState may be simpler
💡 Pro Tip
  • Add a reset() method to return to idle state
  • Extend with retry logic and exponential backoff for resilience
  • Combine with useCallback so execute is stable across renders
