SOLID Principles

Five foundational object-oriented design principles that produce maintainable, flexible, and scalable software systems.

Single Responsibility Open/Closed Liskov Substitution Interface Segregation Dependency Inversion

Coined by Robert C. Martin (Uncle Bob) and popularized by Michael Feathers, SOLID is an acronym for five design principles intended to make software designs more understandable, flexible, and maintainable. While originally formulated for OOP, these principles apply broadly across paradigms.

S · 01
Single Responsibility Principle
A class should have one, and only one, reason to change.
Foundation
What it is
Every class, module, or function should encapsulate exactly one concern. "Reason to change" maps to a single stakeholder or actor whose requirements drive that unit.
Why use it
When a class has multiple responsibilities, a change for one stakeholder can unintentionally break behaviour for another. Isolation prevents that fragility.
Real-world use case
Separating a User domain model from an EmailService. The User class shouldn't know how emails are sent — that's a separate concern.
❌ UserService
→
saves to DB
+
sends email
+
generates PDF
vs
✓ UserRepository
|
EmailService
|
ReportGenerator
TypeScript · srp_example.ts 
// ❌ Violation — UserService has too many responsibilities
class UserService {
  saveUser(user: User) { /* persist to DB */ }
  sendWelcomeEmail(user: User) { /* SMTP logic here */ }
  generateInvoicePdf(user: User) { /* PDF rendering here */ }
}

// ✅ Correct — each class has a single reason to change
class UserRepository {
  save(user: User): Promise<void> {
    return this.db.insert('users', user);
  }
}

class EmailService {
  sendWelcome(user: User): Promise<void> {
    return this.mailer.send({
      to: user.email,
      subject: 'Welcome!',
      template: 'welcome',
    });
  }
}

class InvoiceGenerator {
  generate(user: User): Buffer {
    return this.pdfRenderer.render('invoice', { user });
  }
}
✅ Best PracticeIf you struggle to name a class in one word, it likely has too many responsibilities. Split it.
⚠️ Common MistakeConfusing SRP with "one method per class". SRP is about a single reason to change, not a line count.
💡 Pro TipGroup related methods by the actor they serve. If Finance and HR would both request changes, split the class.
⏭️ When to SkipIn tiny scripts or data-transfer objects (DTOs) with no behaviour, strict SRP separation may add unnecessary indirection.
O · 02
Open/Closed Principle
Open for extension, closed for modification.
Intermediate
What it is
Software entities should be extendable without altering existing code. Achieved through abstractions (interfaces/abstract classes) that allow new behaviour via new implementations.
Why use it
Modifying existing, tested code risks introducing regressions. OCP lets you add functionality by writing new code, leaving proven code untouched.
Real-world use case
A payment processing system where adding Stripe, PayPal, or Crypto means implementing a PaymentProvider interface — the checkout flow never changes.

←implements
StripeProvider
·
PaypalProvider
·

TypeScript · ocp_example.ts 
// ❌ Violation — adding a new type requires editing existing switch
function processPayment(type: string, amount: number) {
  if (type === 'stripe') { /* ... */ }
  else if (type === 'paypal') { /* ... */ }
  // Every new provider = modify this function
}

// ✅ Correct — new providers extend, never modify
interface PaymentProvider {
  charge(amount: number, currency: string): Promise<PaymentResult>;
}

class StripeProvider implements PaymentProvider {
  async charge(amount: number, currency: string) {
    return stripeClient.charges.create({ amount, currency });
  }
}

class CryptoProvider implements PaymentProvider {
  async charge(amount: number, currency: string) {
    return this.wallet.sendTransaction({ amount, currency });
  }
}

// Checkout never needs to change regardless of provider count
class Checkout {
  constructor(private provider: PaymentProvider) {}

  complete(cart: Cart) {
    return this.provider.charge(cart.total, cart.currency);
  }
}
✅ Best PracticeUse the Strategy or Template Method pattern to encapsulate varying behaviour behind a stable interface.
⚠️ Common MistakeOver-engineering: don't abstract prematurely. Apply OCP when you have at least two concrete variations or anticipate real extension points.
💡 Pro TipThe Plugin pattern is OCP in action — a host system defines extension points; plugins provide behaviour without modifying the host.
⏭️ When to SkipPure data structures or configurations where "extension" is just a new field. OCP is most valuable in behavioural, logic-heavy code.
L · 03
Liskov Substitution Principle
Subtypes must be substitutable for their base types without altering program correctness.
Intermediate
What it is
If S is a subtype of T, then objects of type T may be replaced with objects of type S without changing the correctness of the program. Formal contract: preconditions, postconditions, invariants.
Why use it
Violations cause silent bugs when polymorphism is used. LSP ensures inheritance hierarchies are semantically consistent — not just syntactically compatible.
Real-world use case
The classic violation: Square extends Rectangle. Setting width on a Square silently changes height, breaking callers that expect independent dimensions.
Shape (base)
→
Circle.area() ✓
·
Rectangle.area() ✓
·
Square(Rectangle) ✗
TypeScript · lsp_example.ts 
// ❌ Violation — Square breaks Rectangle's invariant
class Rectangle {
  setWidth(w: number)  { this.width = w; }
  setHeight(h: number) { this.height = h; }
  area() { return this.width * this.height; }
}

class Square extends Rectangle {
  setWidth(w: number)  { this.width = this.height = w; }
  setHeight(h: number) { this.width = this.height = h; }
}

function doubleWidth(r: Rectangle) {
  const h = r.height;
  r.setWidth(r.width * 2);
  // Passes Rectangle, fails Square — height was also changed
  console.assert(r.height === h);
}

// ✅ Correct — use a common interface, not inheritance
interface Shape {
  area(): number;
  perimeter(): number;
}

class Rectangle implements Shape {
  constructor(readonly width: number, readonly height: number) {}
  area()      { return this.width * this.height; }
  perimeter() { return 2 * (this.width + this.height); }
}

class Square implements Shape {
  constructor(readonly side: number) {}
  area()      { return this.side ** 2; }
  perimeter() { return 4 * this.side; }
}
✅ Best PracticePrefer composition or shared interfaces over deep inheritance when subtype behavior must diverge from the parent contract.
⚠️ Common MistakeThrowing UnsupportedOperationException (or similar) from an inherited method. That's an LSP violation — the caller's assumptions are broken.
💡 Pro TipDesign by contract: document preconditions (what you require), postconditions (what you guarantee), and invariants. Subtypes may only weaken preconditions and strengthen postconditions.
⏭️ When to SkipLeaf classes with no polymorphic usage. LSP only matters when a reference to the base type is used in a context that may receive a subtype.
I · 04
Interface Segregation Principle
Clients should not be forced to depend on interfaces they do not use.
Foundation
What it is
Large, monolithic interfaces should be split into smaller, role-specific interfaces. Clients only implement what they actually need, preventing "fat interface" pollution.
Why use it
Implementing irrelevant methods leads to empty stubs, thrown exceptions, and tight coupling. Segregated interfaces reduce friction and make dependencies explicit.
Real-world use case
A read-only repository shouldn't be forced to implement save() or delete(). Separate Readable and Writable interfaces solve this.
❌ IWorker: work + eat + sleep
vs
IWorkable: work()
+
IEatable: eat()
+
ISleepable: sleep()
TypeScript · isp_example.ts 
// ❌ Violation — fat interface forces unneeded methods
interface IMultiFunctionPrinter {
  print(doc: Document): void;
  scan(doc: Document): void;
  fax(doc: Document): void;
}

class SimplePrinter implements IMultiFunctionPrinter {
  print(doc: Document) { /* works */ }
  scan(doc: Document)  { throw new Error('Not supported'); } // ← violation
  fax(doc: Document)   { throw new Error('Not supported'); } // ← violation
}

// ✅ Correct — segregated, role-specific interfaces
interface IPrintable { print(doc: Document): void; }
interface IScannable { scan(doc: Document): void; }
interface IFaxable   { fax(doc: Document): void; }

// SimplePrinter only needs IPrintable
class SimplePrinter implements IPrintable {
  print(doc: Document) { /* implementation */ }
}

// Enterprise printer implements all three
class EnterprisePrinter implements IPrintable, IScannable, IFaxable {
  print(doc: Document) { /* ... */ }
  scan(doc: Document)  { /* ... */ }
  fax(doc: Document)   { /* ... */ }
}

// Repository split example
interface ReadRepository<T> {
  findById(id: string): Promise<T | null>;
  findAll(): Promise<T[]>;
}

interface WriteRepository<T> {
  save(entity: T): Promise<void>;
  delete(id: string): Promise<void>;
}
✅ Best PracticeDesign interfaces from the client's perspective, not the implementation's. Ask "what does this consumer actually need?"
⚠️ Common MistakeCreating one interface per method. ISP is about logical cohesion, not granularity for its own sake. Group methods that change together.
💡 Pro TipCQRS naturally enforces ISP — Command and Query interfaces are segregated by definition, making read models lightweight and focused.
⏭️ When to SkipStable, narrow interfaces that all consumers genuinely use. Don't split interfaces that serve a single, coherent contract.
D · 05
Dependency Inversion Principle
High-level modules should not depend on low-level modules. Both should depend on abstractions.
Advanced
What it is
Two rules: (1) High-level policy code must not import concrete low-level details. (2) Abstractions (interfaces) should not depend on details — details should depend on abstractions.
Why use it
Direct coupling to implementations makes testing painful (you can't mock a new MySQLDatabase()) and deployment inflexible. DIP enables swapping implementations at runtime or test-time.
Real-world use case
A NotificationService depending on an IMessageBroker interface rather than a concrete RabbitMQ class. In tests, inject an in-memory mock; in production, inject RabbitMQ.
❌ OrderService → MySQLDB (concrete)
vs
OrderService → IDatabase (abstract)
←
MySQLDB
·
MockDB (test)
TypeScript · dip_example.ts 
// ❌ Violation — OrderService tightly coupled to MySQL
class OrderService {
  private db = new MySQLDatabase(); // ← hardcoded, untestable

  async createOrder(order: Order) {
    await this.db.query(`INSERT INTO orders ...`);
  }
}

// ✅ Correct — depend on abstraction, inject concrete at runtime
interface IOrderRepository {
  save(order: Order): Promise<Order>;
  findById(id: string): Promise<Order | null>;
}

class OrderService {
  // ↓ Receives abstraction via constructor injection
  constructor(private repo: IOrderRepository) {}

  async createOrder(order: Order) {
    this.validateOrder(order);
    return this.repo.save(order);
  }
}

// Production wiring
const service = new OrderService(new MySQLOrderRepository(db));

// Test wiring — no DB needed
const mockRepo: IOrderRepository = {
  save: jest.fn().mockResolvedValue({ id: 'mock-123', ...order }),
  findById: jest.fn().mockResolvedValue(null),
};
const testService = new OrderService(mockRepo);

// Dependency Injection Container (e.g. InversifyJS / tsyringe)
@injectable()
class MySQLOrderRepository implements IOrderRepository {
  constructor(@inject(DATABASE_TOKEN) private db: IDatabase) {}
  async save(order: Order) { /* ... */ }
  async findById(id: string) { /* ... */ }
}
✅ Best PracticeUse constructor injection as the primary DI pattern. It makes dependencies explicit, easy to mock, and impossible to forget at instantiation.
⚠️ Common MistakeConfusing DIP with Dependency Injection. DI is a technique to achieve DIP. You can honor DIP without a DI container by passing interfaces manually.
💡 Pro TipThe interface belongs in the high-level module's package, not the low-level module's. This is the "inversion" — low-level code depends on the high-level module's contracts.
⏭️ When to SkipStable leaf utilities (logging, date formatting) where swapping is implausible. Abstracting truly stable dependencies adds ceremony without value.
