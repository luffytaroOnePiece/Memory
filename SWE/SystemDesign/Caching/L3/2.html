Cache Eviction
Policies

When a cache is full, something must go. Which key you choose to evict determines whether your cache helps or hurts. Master all four policies and know how to implement LRU from scratch.

01 · LRU 02 · LFU 03 · FIFO 04 · Random 05 · LRU Implementation

Every cache has a fixed capacity. When it's full and a new key must be inserted, the cache must evict an existing entry to make room. The eviction policy determines which key is sacrificed. A bad policy evicts hot data, causing a cascade of expensive cache misses. A good policy keeps the most useful data alive, maximizing hit rate.

🎯
Hit Rate
The primary metric. A good policy maximizes the fraction of requests served from cache.
📐
O(1) Operations
Production caches require O(1) get and set — policy overhead cannot dominate request time.
🧠
Access Pattern
The best policy depends on your workload: recency, frequency, or insertion order.
⚖️
Memory Overhead
Tracking frequency costs more memory than tracking recency. Trade-offs matter at scale.

01 — Most Common
LRU — Least Recently Used
The workhorse of production caches. Evict the key you haven't touched in the longest time.
Core Policy
What It Is

Evicts the key whose most recent access is the oldest. The assumption: if you haven't used something recently, you probably won't need it soon. Temporal locality makes this true for most workloads.

Why Use It

Works extremely well for access patterns with temporal locality — users browsing the same pages, the same products trending, the same configs being loaded. It's the sensible default.

Real-World Use Case

A social media feed cache. User @alice's feed is loaded every few seconds. LRU keeps it hot. An old post from 2019 that no one reads gets evicted naturally.

Redis Config

maxmemory-policy allkeys-lru — evicts any key. Or volatile-lru — only evicts keys that have a TTL set.

The Scan Resistance Problem

LRU has one flaw: a sequential scan (e.g., SELECT * over 1M rows) will thrash the cache, evicting hot data to cache cold scan rows that are never re-read.

Redis's Approximated LRU

True LRU requires a doubly linked list over all keys — expensive at millions of keys. Redis uses sampled LRU: on eviction, sample N random keys (default 5), evict the least recently used among them. Good enough at scale.

LRU vs LRU-K

LRU-K evicts the key whose K-th most recent access is oldest. LRU-2 is popular: you need 2 hits to be "promoted" into the main cache, preventing one-hit-wonders from evicting stable hot data.

Clock Algorithm

A practical LRU approximation used in OS page replacement. Each page has a reference bit. On eviction, scan clockwise — if bit=1, clear it (second chance); if bit=0, evict it. O(1), low overhead.

✅ Best default policy
Unless you have specific data showing your workload has non-temporal access, start with LRU. It outperforms FIFO and random on almost every realistic benchmark.
⚠️ Vulnerable to cache pollution
A one-time full table scan will flush all your hot keys. Mitigate with LRU-2, or restrict your cache client to never cache scan-type queries.
💡 Use volatile-lru in Redis for sessions
Only evict keys with TTLs set, so permanent config keys are never evicted. This separates ephemeral cached data from long-lived reference data.
⏭️ Skip LRU for frequency-skewed data
If your top 100 keys get 99% of traffic but LRU keeps cycling through the other 10,000 keys, LFU will dramatically outperform it.
02 — Frequency-Aware
LFU — Least Frequently Used
Evict the key accessed the fewest times. Ideal for highly skewed, Zipfian access distributions.
Advanced
What It Is

Tracks access counts for every key. When eviction is needed, the key with the lowest frequency counter is removed. Keys must earn their place through repeated use.

LFU vs LRU — Key Difference

LRU asks "when was this last used?" LFU asks "how often has this been used overall?" A key accessed 10,000 times but not in the last 5 seconds stays in LFU, but may be evicted by LRU.

Real-World Use Case

A product catalog with power-law access: iPhone Pro is fetched 50,000×/day, while obscure SKUs get 2–3 hits. LFU keeps the iPhone Pro permanently warm; LRU might not.

The Aging Problem

Naïve LFU suffers from "cache pollution by history" — a key that was popular a month ago but is now dead still has a high count and never gets evicted. Solution: frequency decay (halve counts periodically) or use Redis's LFU with configurable decay.

Python — O(1) LFU Cache (Leetcode 460) 
from collections import defaultdict, OrderedDict

class LFUCache:
    """O(1) get and put using two hashmaps + frequency buckets."""

    def __init__(self, capacity: int):
        self.capacity   = capacity
        self.key_val    = {}                    # key → value
        self.key_freq   = {}                    # key → frequency count
        self.freq_keys  = defaultdict(OrderedDict)  # freq → OrderedDict of keys (LRU within freq)
        self.min_freq   = 0                    # current minimum frequency

    def get(self, key: int) -> int:
        if key not in self.key_val:
            return -1
        self._bump_freq(key)
        return self.key_val[key]

    def put(self, key: int, value: int) -> None:
        if self.capacity <= 0: return

        if key in self.key_val:
            self.key_val[key] = value
            self._bump_freq(key)
            return

        if len(self.key_val) >= self.capacity:
            # Evict LRU item from the minimum-frequency bucket
            evict_key, _ = self.freq_keys[self.min_freq].popitem(last=False)
            del self.key_val[evict_key]
            del self.key_freq[evict_key]

        self.key_val[key]  = value
        self.key_freq[key] = 1
        self.freq_keys[1][key] = None
        self.min_freq = 1             # New keys always start at freq 1

    def _bump_freq(self, key: int):
        freq = self.key_freq[key]
        self.key_freq[key] += 1
        del self.freq_keys[freq][key]
        if not self.freq_keys[freq] and freq == self.min_freq:
            self.min_freq += 1         # Advance min_freq if bucket is now empty
        self.freq_keys[freq + 1][key] = None
✅ Use for power-law (Zipfian) access
When a small fraction of keys get the vast majority of traffic, LFU is dramatically superior to LRU. E-commerce product pages, social profiles, media assets.
⚠️ New keys are always eviction candidates
A brand new trending key starts at freq=1 and may be immediately evicted before it can build up frequency. Mitigate with a "new key immunity window" or LFU+LRU hybrid (ARC).
💡 Redis LFU uses logarithmic counters
Redis uses a 8-bit Morris counter — it increments probabilistically to avoid overflow, and decays based on lfu-decay-time. Efficient and surprisingly accurate.
⏭️ Avoid LFU for uniform access
If all keys have similar access frequencies, LFU provides no benefit over LRU and adds memory overhead for tracking counts.
03 — Simplest
FIFO — First In, First Out
Evict whatever was inserted first, regardless of how often it's been used.
Simple
What It Is

A pure queue. Keys enter at the back and leave from the front. The eviction candidate is always the key that has been in the cache the longest — ignoring all usage patterns entirely.

Why It's Suboptimal

FIFO is access-unaware. A key inserted first but still actively used (e.g., a static config) gets evicted while a brand-new key nobody will ever read again stays in cache.

Where It Makes Sense

Stream processing buffers where data is processed in order: message queues, log pipelines, sliding window aggregations. The "oldest" data is genuinely the least useful.

Bélády's Anomaly

FIFO uniquely suffers from Bélády's anomaly: adding more cache capacity can increase miss rate with some access patterns. LRU and LFU are immune to this paradox.

Python — FIFO Cache 
from collections import OrderedDict

class FIFOCache:
    """FIFO cache using OrderedDict (insertion order guaranteed in Python 3.7+)."""

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.store    = OrderedDict()  # maintains insertion order

    def get(self, key: int) -> int:
        # NB: access does NOT change position — FIFO ignores recency
        return self.store.get(key, -1)

    def put(self, key: int, value: int) -> None:
        if key in self.store:
            # Update value but preserve insertion position
            self.store[key] = value
            return

        if len(self.store) >= self.capacity:
            # Evict oldest inserted key (first in OrderedDict)
            self.store.popitem(last=False)

        self.store[key] = value

# Contrast with LRU: in LRU, get() calls move_to_end(key)
# In FIFO, get() does NOT move_to_end — order is purely by insertion.
✅ Perfect for time-ordered streams
Log ingestion pipelines, event queues, sliding time windows. When data genuinely becomes less relevant with age regardless of access, FIFO is semantically correct.
⚠️ Terrible for general-purpose caches
Static configs, user sessions, popular product pages — all get evicted the moment something new is inserted. Hit rates crater compared to LRU on real workloads.
💡 FIFO is the baseline for comparisons
When benchmarking LRU vs LFU, always include FIFO as the naïve baseline. It helps you quantify the benefit of tracking access metadata.
⏭️ Don't use FIFO for user-facing caches
A user loading the same profile page repeatedly won't benefit: their data could be evicted after one read if enough new keys came in, causing repeated DB hits.
04 — Surprisingly Viable
Random Eviction
Pick a random key and evict it. Simpler than you'd think, competitive on certain workloads.
Simple
What It Is

When the cache is full, select a key uniformly at random and evict it. No metadata is tracked. No order is maintained. Pure entropy makes the eviction decision.

When It Competes

On uniform access distributions where all keys are equally likely to be re-requested, there's no information advantage to tracking recency or frequency. Random eviction matches LRU's hit rate while using zero overhead.

Real-World Use Case

CPU TLB (Translation Lookaside Buffer) often uses random replacement. At nanosecond timescales, the overhead of LRU tracking outweighs the marginal hit rate improvement. Hardware prefers simplicity.

Theoretical Bound

For a cache of size k and n total unique keys, a random policy has an expected hit rate proportional to k/n. For a Zipfian distribution (real web traffic), this is far worse than LRU — but for flat distributions, the gap narrows significantly.

Python — Random Eviction Cache 
import random

class RandomCache:
    """Random eviction cache. O(1) get/put with zero metadata overhead."""

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.store    = {}        # key → value
        self.keys     = []        # for O(1) random selection
        self.key_idx  = {}        # key → index in self.keys

    def get(self, key: int) -> int:
        return self.store.get(key, -1)

    def put(self, key: int, value: int) -> None:
        if key in self.store:
            self.store[key] = value
            return

        if len(self.store) >= self.capacity:
            # Pick a random key — swap with last for O(1) removal
            idx    = random.randint(0, len(self.keys) - 1)
            victim = self.keys[idx]

            # Swap victim with last element for O(1) list removal
            last_key              = self.keys[-1]
            self.keys[idx]        = last_key
            self.key_idx[last_key]= idx
            self.keys.pop()

            del self.store[victim]
            del self.key_idx[victim]

        self.store[key]    = value
        self.key_idx[key]  = len(self.keys)
        self.keys.append(key)
✅ Use for truly uniform access patterns
Database buffer pools for OLAP scans with no hot data, or testing/benchmarking where you want a zero-intelligence baseline to compare against.
⚠️ Bad for skewed real-world traffic
Web traffic follows power laws. With random eviction, your most popular page has the same eviction probability as a page nobody visits. Hit rate is proportionally terrible.
💡 Useful in hardware due to zero overhead
CPU caches, TLBs, and branch predictors sometimes use pseudo-random replacement because the logic is simpler than tracking recency at sub-nanosecond timescales.
⏭️ Not for any user-facing application cache
Redis offers allkeys-random, but it's almost never the right choice unless you know your access distribution is flat. Default to LRU instead.
05 — Must Know
Implementing LRU: HashMap + Doubly Linked List
The canonical O(1) LRU implementation. A favorite in system design and coding interviews.
Interview Essential
Why These Two Structures?

HashMap: O(1) key lookup — find a node in the list by key instantly.
Doubly Linked List: O(1) node move — move any node to head (MRU) or remove from tail (LRU) in constant time.

The Core Invariant

The DLL is always sorted by recency: Head = most recently used, Tail = least recently used. On every get or put, the accessed node is moved to the head.

Why Doubly (not Singly) Linked?

To remove an interior node in O(1), you need to update both its predecessor and successor. With only a next pointer, removing a node requires traversal to find the predecessor — making eviction O(n).

Sentinel Nodes Trick

Use fake head and tail sentinel nodes that are never evicted. This eliminates all null pointer edge cases: the list is always non-empty, and you never special-case empty-list operations.

Python — Full LRU Cache — HashMap + Doubly Linked List 
class Node:
    """Doubly linked list node."""
    def __init__(self, key=0, val=0):
        self.key  = key
        self.val  = val
        self.prev = None
        self.next = None


class LRUCache:
    """
    O(1) get and put.
    Structure: HashMap[key → Node] + Doubly Linked List.
    DLL invariant: head.next = MRU, tail.prev = LRU.
    """

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.map      = {}                  # key → Node (O(1) lookup)

        # Sentinel nodes eliminate null-check edge cases
        self.head = Node()                   # dummy head (MRU side)
        self.tail = Node()                   # dummy tail (LRU side)
        self.head.next = self.tail
        self.tail.prev = self.head

    def get(self, key: int) -> int:
        if key not in self.map:
            return -1
        node = self.map[key]
        self._move_to_head(node)              # Mark as most recently used
        return node.val

    def put(self, key: int, value: int) -> None:
        if key in self.map:
            node = self.map[key]
            node.val = value
            self._move_to_head(node)
            return

        node = Node(key, value)
        self.map[key] = node
        self._insert_at_head(node)

        if len(self.map) > self.capacity:
            lru_node = self.tail.prev      # LRU is always just before tail
            self._remove(lru_node)
            del self.map[lru_node.key]

    # ─── DLL helpers ──────────────────────────────────────────

    def _remove(self, node: Node):
        """Unlink node from wherever it is in O(1)."""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _insert_at_head(self, node: Node):
        """Insert node immediately after sentinel head (MRU position)."""
        node.prev       = self.head
        node.next       = self.head.next
        self.head.next.prev = node
        self.head.next  = node

    def _move_to_head(self, node: Node):
        """Re-promote node to MRU position."""
        self._remove(node)
        self._insert_at_head(node)

✅ Sentinel nodes are worth the extra lines
Without sentinels, every insert/remove must null-check head and tail. Sentinels make _remove and _insert_at_head always valid with no conditionals.
⚠️ Common bug: forgetting map deletion on evict
You remove from the DLL but forget del self.map[key]. The node is gone from the list but still reachable from the map — a memory leak and a logical bug.
💡 Python's OrderedDict is a built-in LRU
OrderedDict + move_to_end(key) gives you a clean O(1) LRU in 10 lines. Also: functools.lru_cache for function memoization.
⏭️ In interviews, clarify: get() counts as a use?
Most LRU definitions say yes — a get() promotes the key to MRU. Some variations only promote on put(). State your assumption before coding.
