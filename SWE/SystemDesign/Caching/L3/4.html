Where Caches Live

A technical reference to the four layers of cache deployment — from in-process memory to global edge nodes — and how to choose the right tier for your use case.

01 In-Process / In-Memory 02 Sidecar / Local Redis 03 Distributed Cache 04 CDN / Edge

Choosing where to place a cache is as important as choosing whether to cache. Each tier trades off latency, consistency, memory capacity, and operational complexity. Push the cache closer to the compute, and you get lower latency but harder invalidation. Pull it out to a shared cluster, and you gain consistency but add a network hop.

⚡
Reduce Latency
Serve data from RAM instead of disk or remote services.
📉
Reduce Load
Fewer queries to databases, APIs, and origin servers.
📐
Right-Size the Tier
Match the cache layer to your consistency and scale needs.
🌐
Scale Globally
Edge caches push content physically closer to users.
01 / CACHE TIER

In-Process / In-Memory

Beginner

The fastest cache possible — data lives in the same heap as your application code.

What it is
A data structure (HashMap, LRU map, array) stored inside the process's own memory space. Zero network overhead — a cache hit is just a pointer dereference.
Why use it
Sub-microsecond reads. Perfect for lookup tables, config values, computed constants, or reference data that changes rarely and fits in a few hundred MB.
Real-world use case
A payment service caching country→currency mappings. An auth service caching a parsed public key. A recommendation engine caching a feature weight matrix.

Java · Guava LoadingCache 
import com.google.common.cache.*;

public class UserService {

  private final LoadingCache<Long, User> userCache =
    CacheBuilder.newBuilder()
      .maximumSize(10_000)              // evict when heap fills
      .expireAfterWrite(5, TimeUnit.MINUTES)
      .recordStats()
      .build(key -> db.findUserById(key)); // auto-load on miss

  public User getUser(long userId) {
    return userCache.get(userId); // ~100ns on hit, ~5ms on miss
  }

  public void onUserUpdated(long userId) {
    userCache.invalidate(userId); // must invalidate manually
  }
}
Memory Pressure
All cached data lives on the JVM/Go/Node.js heap. Large caches trigger GC pauses in managed runtimes. Size limits (e.g., maximumSize or maximumWeight in Guava/Caffeine) are mandatory — not optional.
Consistency Problem
Each process instance holds its own independent copy. In a 3-replica deployment, a write invalidating one replica's cache leaves the other two stale until TTL expiry. Known as "thundering herd" when all miss simultaneously.
Caffeine vs Guava
Prefer Caffeine for JVM services — it uses a lock-free W-TinyLFU eviction policy that outperforms Guava's LRU by 2–5× in hit rate for skewed workloads. Go: use sync.Map or ristretto.
Go · sync.Map + TTL expiry 
type entry struct {
    value    any
    expiresAt time.Time
}

type TTLCache struct { m sync.Map }

func (c *TTLCache) Set(key string, val any, ttl time.Duration) {
    c.m.Store(key, entry{val, time.Now().Add(ttl)})
}

func (c *TTLCache) Get(key string) (any, bool) {
    v, ok := c.m.Load(key)
    if !ok { return nil, false }
    e := v.(entry)
    if time.Now().After(e.expiresAt) {
        c.m.Delete(key) // lazy eviction
        return nil, false
    }
    return e.value, true
}
✅ Performance TipUse Caffeine (Java) or ristretto (Go) for production — they have sophisticated eviction policies, async refresh, and per-entry weights that outperform naive HashMap wrappers.
⚠️ Common MistakeForgetting to set a size cap. An unbounded in-process cache will slowly consume all available heap, causing OOM errors or degraded GC performance under load.
💡 Pro TipUse refreshAfterWrite (Caffeine) instead of expireAfterWrite for hot data — the first stale reader triggers an async reload while still returning the old value, eliminating cold-miss spikes.
⏭️ When to SkipSkip in-process caching when data changes frequently across multiple writers, or when running many replicas where stale reads would cause correctness issues (e.g., billing, inventory counts).
02 / CACHE TIER

Sidecar / Local Redis

Intermediate

A dedicated cache process co-located on the same host — separate memory, single network hop.

What it is
A Redis or Memcached instance running as a sidecar container or separate systemd process on the same VM/pod as your app. Communication is over localhost (127.0.0.1), not a remote network.
Why use it
Data survives process restarts, can be shared between threads without locking, and the cache can hold significantly more data than heap allows — with richer data structures (sorted sets, HyperLogLog).
Real-world use case
A video streaming service with one Redis sidecar per encoding worker caching segment manifests. Each worker reads its own local cache before hitting the object store, reducing cold reads by 80%.

Python · redis-py with cache-aside 
import redis, json

# Connects to sidecar on localhost — sub-millisecond latency
r = redis.Redis(host="127.0.0.1", port=6379, decode_responses=True)

def get_product(product_id: int) -> dict:
    key = f"product:{product_id}"
    cached = r.get(key)

    if cached:
        return json.loads(cached)  # cache hit — ~0.3ms

    # Cache miss — fetch from DB and populate
    product = db.query("SELECT * FROM products WHERE id = %s", product_id)
    r.setex(key, 300, json.dumps(product))  # TTL = 5 min
    return product

def update_product(product_id: int, data: dict):
    db.update(product_id, data)
    r.delete(f"product:{product_id}")  # invalidate on write
Sidecar vs Shared
A sidecar Redis is not shared across instances. This avoids network latency but means each node has its own cold cache on startup and independent data — you lose cross-instance cache hits.
Kubernetes Pattern
In K8s, add Redis as a second container in the same Pod spec. They share network namespace — localhost works. Mount a ConfigMap with maxmemory-policy allkeys-lru to handle eviction under pressure.
Memcached vs Redis
Use Memcached for pure key-value caching at extreme throughput — it's simpler and slightly faster for string reads. Use Redis when you need TTL precision, pub/sub invalidation, sorted sets, or persistence.
YAML · Kubernetes Pod with Redis sidecar 
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: app
      image: myapp:latest
      env:
        - name: REDIS_URL
          value: "redis://127.0.0.1:6379" # localhost — no DNS needed

    - name: redis-sidecar
      image: redis:7-alpine
      args: ["--maxmemory", "256mb", "--maxmemory-policy", "allkeys-lru"]
      resources:
        limits:
          memory: "280Mi"  # slightly above maxmemory
✅ Performance TipSet maxmemory-policy allkeys-lru explicitly — the default Redis policy is noeviction, which causes errors when full instead of evicting old keys.
⚠️ Common MistakeTreating sidecar Redis like a durable store. A pod restart flushes all sidecar state. Never store write-ahead data or session tokens here without replication.
💡 Pro TipPipeline multiple Redis commands with MULTI/EXEC or client-side pipelining to batch round-trips — even on localhost, batching 10 commands in one pipeline cuts overhead by ~8×.
⏭️ When to SkipSkip sidecar caches when you need cross-instance cache sharing — e.g., user session stores, shared rate-limit counters, or leaderboard state. Use a distributed cache instead.
03 / CACHE TIER

Distributed Cache

Intermediate–Advanced

A shared cache cluster accessible to all service instances — consistent reads, higher latency than local.

What it is
A standalone Redis Cluster, Redis Sentinel, or Memcached cluster shared across all pods/instances of a service. Every instance reads from and writes to the same cache — one source of truth.
Why use it
All instances share warmed cache entries immediately. Rate limiters, session stores, distributed locks (Redlock), and shopping carts require a single consistent cache visible to all replicas.
Real-world use case
An e-commerce checkout service running 20 replicas uses a Redis Cluster for cart state. Any replica can read any user's cart — session affinity is not required.

Node.js · ioredis Cluster + rate limiter 
import Redis from 'ioredis';

const cluster = new Redis.Cluster([
  { host: 'redis-0.redis-headless', port: 6379 },
  { host: 'redis-1.redis-headless', port: 6379 },
  { host: 'redis-2.redis-headless', port: 6379 },
]);

// Sliding window rate limiter — works across ALL replicas
async function rateLimit(userId: string): Promise<boolean> {
  const key = `rate:${userId}`;
  const now = Date.now();
  const window = 60_000; // 1 minute

  const [, count] = await cluster
    .multi()
    .zremrangebyscore(key, 0, now - window)
    .zadd(key, now, `${now}`)
    .zcard(key)
    .expire(key, 60)
    .exec();

  return (count as number) <= 100; // 100 req/min per user
}
Consistent Hashing
Redis Cluster uses 16,384 hash slots. Each key maps to a slot via CRC16(key) mod 16384. Adding or removing nodes only remaps a fraction of slots — avoiding a full rehash storm seen in naive modulo sharding.
Hot Key Problem
A single key receiving millions of reads (e.g., a viral post) can saturate one shard's CPU. Mitigate with local read-through replicas, key sharding suffixes (key:{0..N}), or client-side in-process L1 caches for top-N hot keys.
Cache Stampede
When a TTL expires simultaneously for a hot key, all instances miss and flood the origin. Use probabilistic early expiration (PER algorithm) or a distributed mutex (SET NX PX) to let one instance recompute while others wait.
Python · Distributed lock + stampede prevention 
import uuid, time

def get_with_lock(r, key, fetch_fn, ttl=300):
    # Try cache first
    value = r.get(key)
    if value:
        return value

    lock_key = f"lock:{key}"
    lock_id = str(uuid.uuid4())

    # Acquire mutex — only one process recomputes
    acquired = r.set(lock_key, lock_id, nx=True, ex=10)

    if acquired:
        try:
            result = fetch_fn()
            r.setex(key, ttl, result)
            return result
        finally:
            # Release only our lock (atomic Lua script)
            r.eval("""
              if redis.call('get',KEYS[1])==ARGV[1]
              then return redis.call('del',KEYS[1])
              else return 0 end""",
              1, lock_key, lock_id)
    else:
        time.sleep(0.05)          # wait and retry
        return r.get(key)       # should be populated now
✅ Performance TipUse connection pooling aggressively — each new Redis connection costs ~1ms. With 20 app replicas each opening 10 connections, you get 200 persistent connections that amortize handshake overhead.
⚠️ Common MistakeUsing KEYS * in production. It blocks the Redis event loop for the full scan duration — use SCAN with a cursor instead, which is non-blocking and cursor-safe.
💡 Pro TipUse Redis hash tags ({user}.cart) to force related keys onto the same slot, enabling atomic multi-key operations via MULTI/EXEC without cross-slot errors in cluster mode.
⏭️ When to SkipSkip distributed caches for very large binary blobs (>1MB per key) — use object storage with signed URLs + CDN instead. Redis is optimized for small-to-medium serialized objects.
04 / CACHE TIER

CDN / Edge Caching

Advanced

Cache at the network edge — physically close to users, globally distributed, with zero origin load on hits.

What it is
A Content Delivery Network caches HTTP responses at Points of Presence (PoPs) distributed globally. Requests are routed to the nearest PoP via Anycast. Cache misses are forwarded to the origin server.
Why use it
Reduces round-trip time from 200ms (cross-continent) to 10–20ms (local PoP). Absorbs traffic spikes — a viral event hitting 1M RPS never touches your origin if cache hit rate is high.
Real-world use case
A media site serving article HTML with Cache-Control: public, s-maxage=600. Cloudflare caches the page at ~300 PoPs. 99% of readers hit edge cache — origin only serves cache-miss refreshes.

HTTP · Cache-Control header patterns 
# Static assets with content hash in filename (e.g., app.a3f9b2.js)
Cache-Control: public, max-age=31536000, immutable
# CDN and browser cache forever — safe because the hash changes on deploy

# Frequently updated HTML page — CDN caches for 10 min, browser must revalidate
Cache-Control: public, s-maxage=600, max-age=0, must-revalidate
Surrogate-Key: article-42 author-7  # Fastly tag-based purge

# User-specific data — CDN must NOT cache, browser may cache briefly
Cache-Control: private, max-age=60
Vary: Authorization

# API response — CDN-cacheable, stale-while-revalidate for smooth UX
Cache-Control: public, s-maxage=30, stale-while-revalidate=300
ETag: "a3f9b2c8"  # allows conditional GET (304 Not Modified)
Purge Strategies
Tag-based purge (Fastly, Cloudflare) lets you invalidate all pages associated with a resource in one API call. URL-based purge is simpler but doesn't scale when one content change affects thousands of pages.
stale-while-revalidate
Instructs the CDN to serve a stale (expired) cached response immediately while fetching a fresh copy in the background. Eliminates synchronous origin round-trips on cache expiry — users never see the latency spike.
Edge Compute
Cloudflare Workers, Fastly Compute, and Vercel Edge Functions run code at PoPs. You can personalize cached content, implement A/B tests, or run auth checks at the edge without hitting origin — sub-5ms globally.
JavaScript · Cloudflare Worker — serving stale + background refresh 
addEventListener('fetch', event => {
  event.respondWith(handleRequest(event));
});

async function handleRequest(event) {
  const cache = await caches.open('v1');
  const cached = await cache.match(event.request);

  if (cached) {
    const age = Date.now() - new Date(cached.headers.get('Date'));

    if (age > 30_000) {
      // Stale — revalidate in background, return stale immediately
      event.waitUntil(refreshCache(cache, event.request));
    }
    return cached; // always respond fast
  }

  return refreshCache(cache, event.request);
}

async function refreshCache(cache, request) {
  const response = await fetch(request);
  if (response.ok) cache.put(request, response.clone());
  return response;
}
✅ Performance TipUse fingerprinted asset URLs (bundle.a3f9b2.js) and set immutable in Cache-Control. This lets both CDN and browser cache indefinitely, while new deploys automatically bust the cache via the new hash.
⚠️ Common MistakeCaching responses that contain the Set-Cookie header. CDNs that cache cookies will serve one user's session cookie to another user — a critical security bug. Always set Vary: Cookie or strip cookies at the edge.
💡 Pro TipUse Surrogate-Key (Fastly) or Cache-Tag (Cloudflare) to group related pages. When article #42 is updated, purge the tag article-42 to instantly invalidate every page that embeds it — no URL enumeration required.
⏭️ When to SkipSkip CDN caching for highly personalized or private responses (dashboards, account pages). The Vary: Authorization header effectively disables CDN caching — use a distributed cache server-side instead.
