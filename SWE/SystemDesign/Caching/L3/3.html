Caching Patterns

Four foundational strategies for integrating a cache layer between your application and database — with tradeoffs, use cases, and working code.

→ Cache-Aside → Read-Through → Write-Through → Write-Back → Compare All

Why caching patterns matter

A cache is only as effective as how you keep it synchronized with your source of truth. Each pattern makes different tradeoffs between consistency, complexity, and throughput. Choosing the wrong pattern can lead to stale data bugs or wasted write overhead that negates your performance gains.

PATTERN 01
Cache-Aside
The application manages the cache explicitly — lazy loading on demand
MOST COMMON
What it is

The app checks the cache first. On a miss, it fetches from the DB and writes the result back into the cache for future requests.

Why use it

Only loads data that is actually requested. The cache never stores data nobody needs — great for large datasets with skewed access.

Real-world use case

User profile pages on a social platform. Most profiles are cold; only popular or recently active ones stay warm in cache.

App

Cache
HIT ↓ MISS →
skip DB
DB

Cache
Thundering Herd

A cache miss under heavy load causes many concurrent requests to hit the DB simultaneously. Mitigate with mutex locks or probabilistic early expiry.

Stale Data Window

Data updated in the DB is not immediately reflected in the cache. Use TTL + explicit invalidation on writes to bound staleness.

Cold Start Problem

After a restart or cache flush, every request is a miss. Pre-warm the cache with top-N records during deploy or use a persistent cache layer.

PYTHON · CACHE-ASIDE WITH REDIS 
1import json


2import redis


3


4cache = redis.Redis(host="localhost", port=6379, decode_responses=True)


5TTL = 300  # 5 minutes


6


7def get_user(user_id: int, db_session) -> dict:


8    key = f"user:{user_id}"


9


10    # 1. Check cache first


11    cached = cache.get(key)


12    if cached:


13        return json.loads(cached)  # Cache HIT


14


15    # 2. Cache MISS — fetch from database


16    user = db_session.query(User).filter_by(id=user_id).first()


17    if not user:


18        return None


19


20    # 3. Populate cache for subsequent reads


21    user_data = user.to_dict()


22    cache.setex(key, TTL, json.dumps(user_data))


23    return user_data


24


25def update_user(user_id: int, data: dict, db_session) -> None:


26    # Write to DB first, then invalidate cache


27    db_session.query(User).filter_by(id=user_id).update(data)


28    db_session.commit()


29    cache.delete(f"user:{user_id}")  # Invalidate stale entry
✅ Performance tipUse pipeline/batch gets in Redis to fetch multiple keys in a single round-trip for list views.
⚠️ Common mistakeForgetting to invalidate the cache on writes. Stale reads will persist until TTL expiry.
💡 Pro tipAdd jitter to TTLs (e.g., base ± 30s) to prevent cache stampedes when many keys expire at once.
⏭️ When to skipWhen you need guaranteed consistency — a missed invalidation means stale reads until TTL.
PATTERN 02
Read-Through
The cache itself fetches missing data from the DB — transparent to the application
INTERMEDIATE
What it is

The application always reads from the cache. When the cache misses, it automatically loads data from the DB and stores it — the app never touches the DB directly.

Why use it

Keeps the caching logic centralized in the cache layer rather than scattered in every service. Simpler application code at the cost of a smarter cache.

Real-world use case

CDN edge caches: the edge node fetches the origin on a miss and serves all future requests for that asset — the client always talks to the CDN.

App
→always reads
Cache Layer
→on miss only
DB
→auto-populate
Cache Layer
Requires Smart Cache

The cache provider must support a configurable "loader" callback. Redis alone doesn't do this — you need a library like Spring Cache, Guava, or Caffeine.

First-Request Penalty

First access is always a cache miss with full DB latency. Consider pre-warming on startup or combining with write-through to ensure initial data is already cached.

vs. Cache-Aside

Read-through is conceptually identical to cache-aside — the key difference is who populates the cache: the cache library (read-through) vs. the application (cache-aside).

JAVA · READ-THROUGH WITH SPRING CACHE 
1// Spring's @Cacheable = read-through pattern


2// Cache populates automatically on miss — no manual logic needed


3


4@Service


5public class UserService {


6


7    @Autowired


8    private UserRepository repo;


9


10    // Cache key: "users::42"


11    @Cacheable(value = "users", key = "#userId")


12    public User getUser(Long userId) {


13        // Only called on cache miss — framework handles the rest


14        return repo.findById(userId)


15                   .orElseThrow(() -> new UserNotFoundException(userId));


16    }


17


18    @CacheEvict(value = "users", key = "#user.id")


19    public User updateUser(User user) {


20        return repo.save(user);  // Evicts stale cache entry on save


21    }


22}
✅ Performance tipCombine with write-through so the cache is always warm on the first read after a write.
⚠️ Common mistakeCaching null results from the DB. A missing key will hammer the DB on every request — use a sentinel value.
💡 Pro tipUse @CachePut over @CacheEvict after writes to avoid a cold-miss penalty on the next read.
⏭️ When to skipIf your cache layer doesn't support custom loaders. Without that, you're just doing cache-aside in a trenchcoat.
PATTERN 03
Write-Through
Every write goes to both the cache and DB simultaneously — consistency first
INTERMEDIATE
What it is

Writes are synchronously applied to both cache and database. The operation only completes when both stores have acknowledged the write.

Why use it

Cache is never stale — reads after a write always get fresh data. Eliminates the invalidation problem entirely for write paths.

Real-world use case

Financial account balances or inventory counts where a stale read could have real consequences — correctness outweighs write latency.

App
→writes to both
Cache
DB
←both ack
App ✓
Write Amplification

Every write hits two stores, increasing write latency and load. For write-heavy workloads this overhead can negate read benefits — profile first.

Wasted Cache Space

Data written once and never read again still occupies cache memory. Pair with TTL-based eviction to reclaim space for cold write paths.

Partial Failure

If DB write succeeds but cache write fails, or vice versa, you have inconsistency. Wrap in a transaction or use optimistic locking strategies.

TYPESCRIPT · WRITE-THROUGH WITH REDIS 
1import { createClient } from "redis";


2import { db } from "./database";


3


4const cache = createClient();


5const TTL = 600; // 10 minutes


6


7// Write-through: both stores updated before returning


8async function updateInventory(


9  itemId: string,


10  quantity: number


11): Promise<void> {


12  const key = `inventory:${itemId}`;


13


14  // Both writes in parallel — only return when both succeed


15  await Promise.all([


16    cache.setEx(key, TTL, String(quantity)),


17    db.execute(


18      "UPDATE inventory SET qty = $1 WHERE id = $2",


19      [quantity, itemId]


20    ),


21  ]);


22}


23


24async function getInventory(itemId: string): Promise<number> {


25  const cached = await cache.get(`inventory:${itemId}`);


26  // Since write-through guarantees freshness, always trust cache


27  return cached !== null ? parseInt(cached) : fetchFromDB(itemId);


28}
✅ Performance tipRun cache and DB writes in parallel with Promise.all to avoid double-latency on the write path.
⚠️ Common mistakeUsing write-through for write-heavy workloads with no benefit from caching reads — you pay double write cost for nothing.
💡 Pro tipCombine with read-through: writes go through write-through, reads go through read-through. Fully transparent to the app layer.
⏭️ When to skipHigh write throughput use cases — the doubled write latency will become a bottleneck. Use write-back instead.
PATTERN 04
Write-Back / Write-Behind
Writes hit the cache immediately; DB is updated asynchronously in the background
COMPLEX
What it is

Writes are acknowledged as soon as they land in the cache. A background worker (or queue) asynchronously flushes dirty cache entries to the database.

Why use it

Write latency approaches cache speed (microseconds) instead of DB speed (milliseconds). Burst writes are absorbed and batched, reducing DB write amplification.

Real-world use case

Gaming leaderboards or view counters. Thousands of score updates per second hit cache; DB is updated every few seconds with the latest aggregated values.

App
→instant ack
Cache
→async / batch
Queue
→flush
DB
Data Loss Risk

If the cache node crashes before flushing, unflushed writes are lost. Mitigate with AOF persistence in Redis or by writing to a durable write-ahead log first.

Dirty Tracking

You must track which cache entries are "dirty" (unflushed). Redis doesn't do this natively — implement via a secondary set of dirty keys or a message queue (Kafka, SQS).

Operational Complexity

You need a reliable background worker with retry logic, dead-letter handling, and monitoring. A bug in the flusher can silently drop writes.

PYTHON · WRITE-BACK WITH DIRTY KEY TRACKING 
1import threading, time, json


2import redis


3


4cache = redis.Redis(decode_responses=True)


5DIRTY_SET = "dirty_keys"  # Redis set tracking unflushed keys


6FLUSH_INTERVAL = 5          # Flush every 5 seconds


7


8def write_score(user_id: str, score: int) -> None:


9    key = f"score:{user_id}"


10    # Step 1: Write to cache atomically, mark dirty


11    pipe = cache.pipeline()


12    pipe.set(key, score)


13    pipe.sadd(DIRTY_SET, key)   # Mark for background flush


14    pipe.execute()             # Atomic — returns immediately


15


16def background_flusher(db_session) -> None:


17    """Runs in a separate thread, periodically syncs to DB."""


18    while True:


19        time.sleep(FLUSH_INTERVAL)


20        # Step 2: Pop all dirty keys and flush to DB


21        dirty_keys = cache.smembers(DIRTY_SET)


22        for key in dirty_keys:


23            value = cache.get(key)


24            if value is not None:


25                user_id = key.split(":")[1]


26                db_session.execute(


27                    "UPDATE scores SET val=$1 WHERE uid=$2",


28                    [value, user_id]


29                )


30        cache.srem(DIRTY_SET, *dirty_keys)  # Clear after successful flush
✅ Performance tipBatch your DB flushes using INSERT ... ON CONFLICT DO UPDATE (upsert) to reduce round trips to a single statement.
⚠️ Common mistakeNot handling the crash-between-cache-write-and-flush scenario. Always use Redis persistence (AOF) or a durable queue as the write-ahead log.
💡 Pro tipUse Kafka as the flush queue — it gives you durability, replay, and consumer lag monitoring out of the box.
⏭️ When to skipAny use case where losing in-flight writes is unacceptable (payments, auth tokens). The async flush window is a real durability gap.

Pattern Comparison

Pattern Consistency Write Latency Read Latency Complexity Best for
Cache-Aside Eventual Fast (DB only) Fast (on hit) Low General-purpose reads
Read-Through Eventual Fast (DB only) Fast (on hit) Medium Transparent caching
Write-Through Strong Slow (2× writes) Fast Medium Consistency-critical reads
Write-Back Weak Very fast Fast High High write throughput
💡 Exam tip: Cache-aside is the most commonly tested and deployed pattern. Know that it risks stale reads between a DB update and TTL expiry, and that the fix is explicit cache invalidation on writes — not just relying on TTL.
