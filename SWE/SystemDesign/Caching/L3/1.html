Caching Core Concepts

A complete reference for how caches work — from latency reduction to TTL strategies. Master these before your next system design interview.

01 · What is a Cache 02 · Hit vs Miss 03 · Hot & Cold Data 04 · TTL 05 · Read Path

A cache is a high-speed data storage layer that holds a subset of data so that future requests for that data are served faster than is possible by accessing the primary storage layer. Caches sit between the consumer (a service, a user) and the source of truth (a database, an API), absorbing repeated reads.

⚡
Latency
Memory reads are ~100× faster than disk; Redis RTT <1ms vs DB at 5–50ms+
🛡️
DB Offload
Shields your database from thundering-herd reads; fewer connections, lower cost
📈
Throughput
In-memory stores handle millions of ops/sec — orders of magnitude beyond RDBMS
💰
Cost
Reduce expensive query compute, egress fees, and DB read replicas
01 — Foundation
What is a Cache?
A fast, intermediate store between your application and its data source
Fundamental
What It Is

A key-value store (like Redis or Memcached) that sits in front of your database. Frequently-read data is copied into the cache so future reads skip the DB entirely.

Why Use It

Database queries often take 5–100ms. An in-memory lookup takes <1ms. For read-heavy workloads, even a 30% hit rate dramatically reduces average latency.

Real-World Use Case

Twitter caches the home timeline in Redis. Each timeline load reads from cache instead of joining millions of tweet rows per user request.

Where Caches Live

In-process (local memory), distributed (Redis cluster), CDN edge (static assets), CPU L1/L2/L3 (hardware) — each with different latency and scope tradeoffs.

Cache Topology

Local cache: in-process HashMap. Zero network overhead, but not shared across instances. Suitable for static config or small lookup tables.

Distributed Cache

Redis / Memcached: shared across all app servers. One node warms the cache for all. Requires network hop (~0.5ms LAN), but vastly beats DB RTT.

Eviction Policies

LRU (Least Recently Used) is most common — discards the key not accessed for the longest time. LFU (Least Frequently Used) better for non-temporal access patterns.

Cache Sizing

Start with 20–30% of your working dataset. Monitor your hit rate — below 80% means either cache is too small or you're caching the wrong data.

Python — Cache-Aside Pattern 
import redis
import json

redis_client = redis.Redis(host='localhost', port=6379, decode_responses=True)

def get_user_profile(user_id: int) -> dict:
    cache_key = f"user:profile:{user_id}"

    # 1. Try the cache first
    cached = redis_client.get(cache_key)
    if cached:
        return json.loads(cached)  # Cache HIT — fast path ⚡

    # 2. Cache MISS — fall back to the database
    user = db.query("SELECT * FROM users WHERE id = %s", user_id)

    if user:
        # 3. Populate the cache for future reads (TTL = 300s)
        redis_client.setex(cache_key, 300, json.dumps(user))

    return user
✅ Cache read-heavy, rarely-changing data
User profiles, product listings, config flags — data that's read 100× more than it's written.
⚠️ Don't cache unique or one-off queries
Caching SELECT * WHERE timestamp > now() gives 0% hit rate and wastes memory.
💡 Use a consistent key naming scheme
entity:id:field — e.g. user:42:profile. Prevents key collisions and aids debugging.
⏭️ Skip the cache for financial/audit data
Balances, transactions, and compliance logs must always reflect the latest DB state. Stale reads here have real consequences.
02 — Mechanics
Cache Hit vs Cache Miss
Two outcomes, vastly different performance implications
Fundamental
Cache Hit ✅

The requested key exists in the cache and hasn't expired. The data is returned immediately — no database query is made. Typical latency: < 1ms.

Cache Miss ❌

The key is absent (never written, was evicted, or expired). The app falls through to the database, then typically writes the result back to cache. Typical latency: 5–100ms.

Python — Hit Rate Instrumentation 
from dataclasses import dataclass, field
import time

@dataclass
class CacheStats:
    hits: int = 0
    misses: int = 0

    @property
    def hit_rate(self) -> float:
        total = self.hits + self.misses
        return self.hits / total if total > 0 else 0.0

stats = CacheStats()

def get_with_stats(key: str):
    start = time.perf_counter()
    value = redis_client.get(key)
    latency_ms = (time.perf_counter() - start) * 1000

    if value:
        stats.hits += 1
        print(f"HIT  [{latency_ms:.2f}ms] key={key}")
    else:
        stats.misses += 1
        print(f"MISS [{latency_ms:.2f}ms] key={key}")

    return value

# Target: hit_rate >= 0.80 for most read-heavy services
print(f"Hit rate: {stats.hit_rate:.1%}")
✅ Monitor hit rate in production
Redis exposes INFO stats with keyspace_hits and keyspace_misses. Alert if hit rate drops below your baseline.
⚠️ Cold start = all misses
After a deploy or Redis restart, the cache is empty. Expect a latency spike until the cache warms up. Consider pre-warming critical keys on startup.
💡 Cold cache ≠ broken system
A miss just means a slightly slower path. Your app must always handle misses gracefully — the DB is your source of truth, the cache is a hint.
⏭️ Don't obsess over 100% hit rate
A 90% hit rate means 10% of requests hit the DB. That's typically fine. Chasing 100% often means over-caching stale or rarely-needed data.
03 — Data Patterns
Hot Data vs Cold Data
Not all data deserves a cache entry — understand what's worth storing
Intermediate
Hot Data 🔥

Accessed frequently by many requests. High read-to-write ratio. Examples: home feeds, trending items, global config, rate limit counters. This data has high cache utility.

Cold Data 🧊

Rarely accessed, or highly unique per request. Examples: archived records, audit logs, one-time report exports. Caching cold data wastes memory and lowers hit rate.

Identifying Hot Data

Instrument your DB query logs. Look for the top N most-executed queries by volume. The tables and rows they touch are your hot data. Also consider Pareto's principle: 80% of traffic often touches 20% of data.

Skewed Access (Hotspots)

A single extremely hot key (e.g. a celebrity profile) can overwhelm a single cache shard. Solutions: key replication across shards, local in-process caching, or CDN edge caching for public data.

Python — Tiered Caching by Access Pattern 
# Hot data: short TTL, always cached, high churn tolerated
def get_trending_posts(region: str) -> list:
    key = f"trending:{region}"
    cached = redis_client.get(key)
    if cached:
        return json.loads(cached)

    posts = db.query("SELECT * FROM posts ORDER BY views DESC LIMIT 20")
    redis_client.setex(key, 60, json.dumps(posts))  # 60s TTL
    return posts

# Warm data: longer TTL, invalidated on write
def get_user_profile(user_id: int) -> dict:
    key = f"user:profile:{user_id}"
    cached = redis_client.get(key)
    if cached:
        return json.loads(cached)

    user = db.query("SELECT * FROM users WHERE id = %s", user_id)
    redis_client.setex(key, 3600, json.dumps(user))  # 1h TTL
    return user

# Cold data: skip the cache entirely
def get_audit_log(entry_id: int) -> dict:
    # Audit logs are unique, rarely re-read — go straight to DB
    return db.query("SELECT * FROM audit_log WHERE id = %s", entry_id)
✅ Profile before caching
Run EXPLAIN ANALYZE on your heaviest queries. Cache the hot tables, not everything.
⚠️ Beware cache hotspot (hot key)
If 1 key gets 10,000 req/s, one Redis node becomes a bottleneck. Use local_cache[key] as an L1 in front of Redis.
💡 Cache stampede on hot data
When a popular key expires, thousands of requests hit the DB simultaneously. Use probabilistic early expiration or a mutex lock to prevent it.
⏭️ Don't cache user-specific aggregate data
user:123:recommendations has 0% reuse outside that user — you'd cache N entries for N users with near-zero hit rate benefit.
04 — Expiration
TTL — Time-To-Live
Every cached value needs a sunset — learn to set the right one
Intermediate
What TTL Does

A TTL is a counter attached to a cache key. When it reaches zero, the key is automatically deleted. This prevents serving stale data indefinitely and bounds your cache's memory footprint.

Why It Matters

Without TTLs, your cache fills with stale entries forever. With TTLs, you get eventual consistency: the DB is always authoritative, and the cache converges within TTL seconds of any write.

TTL Too Short

Keys expire before they can be reused. Hit rate drops. Your DB sees more load than expected. The cache becomes almost pointless for anything with moderate traffic.

TTL Too Long

Stale data persists. A user updates their profile but sees the old version for hours. Can cause correctness bugs in financial, inventory, or user-facing data.

TTL Reference Guide
30–60s
Trending / real-time feeds
5–60 min
User profiles, product listings
1–24 hrs
Static config, feature flags
Python — TTL Strategies 
import random

# Strategy 1: Fixed TTL — simplest, works for most cases
redis_client.setex("user:42", 300, data)   # Expires in 300s

# Strategy 2: Jittered TTL — prevents cache stampede
# Many keys with the same TTL expire simultaneously → DB spike
base_ttl = 300
jitter = random.randint(-30, 30)  # ±10% variation
redis_client.setex("trending", base_ttl + jitter, data)

# Strategy 3: Write-through invalidation — TTL as a safety net
def update_user_profile(user_id: int, updates: dict):
    # 1. Write to DB (source of truth)
    db.update("UPDATE users SET ... WHERE id = %s", user_id)

    # 2. Immediately invalidate (delete) the cache key
    # Don't update it — let the next read re-populate with fresh data
    redis_client.delete(f"user:profile:{user_id}")

# Strategy 4: TTL check — inspect remaining TTL on a key
remaining = redis_client.ttl("user:42")
# -2 = key doesn't exist, -1 = key has no TTL set (infinite)
if remaining == -1:
    print("WARNING: cache key has no TTL — memory leak risk")
✅ Add jitter to batch-set TTLs
If 10,000 keys all expire at T+300s, your DB gets slammed. Add ±10% random jitter to spread expirations.
⚠️ Never set TTL = -1 (no expiry) blindly
Only omit TTL for data that never changes (e.g. a country code lookup table). Otherwise you risk serving stale data forever.
💡 Combine TTL with active invalidation
Use TTL as a safety net, but also delete/update the key immediately on writes. This gives you both consistency and resilience.
⏭️ Don't set a single global TTL
Different data has different staleness tolerance. User names can be cached for an hour. Stock prices for 1 second. Tune per data type.
05 — Architecture
The Full Read Path
Trace a request from client to cache to DB and back — know every step
Advanced
Cache-Aside (Lazy Loading)

The most common pattern. The app manages the cache: on read, check cache first. On miss, hit DB, then write to cache. The cache is only populated on demand.

Read-Through

The cache itself fetches from the DB on a miss. The app only talks to the cache. This hides DB access details from the app layer — useful with cache providers like Ehcache or AWS ElastiCache.

Python — Complete Cache-Aside Read Path 
from typing import Optional
import json, redis, logging

log = logging.getLogger(__name__)
cache = redis.Redis(host='redis', port=6379, decode_responses=True)

def get_user(user_id: int) -> Optional[dict]:
    """Full cache-aside read path with observability."""
    cache_key = f"user:v2:{user_id}"  # Versioned key for schema changes
    TTL_SECONDS = 300

    # Step 1: Attempt cache lookup
    try:
        raw = cache.get(cache_key)
    except redis.RedisError as e:
        # Step 1a: Cache is unavailable — degrade gracefully, go to DB
        log.warning(f"Redis error: {e}, falling back to DB")
        raw = None

    if raw is not None:
        # Step 2a: CACHE HIT — deserialize and return
        log.debug(f"Cache HIT: {cache_key}")
        return json.loads(raw)

    # Step 2b: CACHE MISS — query the database
    log.debug(f"Cache MISS: {cache_key}")
    user = db.query_one("SELECT * FROM users WHERE id = %s", user_id)

    if user is None:
        # Step 3: User doesn't exist — cache negative result briefly
        # Prevents DB hammering for non-existent IDs (negative caching)
        cache.setex(cache_key, 30, json.dumps(None))
        return None

    # Step 4: Populate cache for future reads
    cache.setex(cache_key, TTL_SECONDS, json.dumps(user))
    return user
✅ Always handle Redis being down
Wrap cache ops in try/except. If Redis fails, fall through to the DB. Your app should degrade, not crash.
⚠️ Cache negative results too
If someone queries user:99999 (nonexistent) 10,000 times, every request hits the DB. Cache null for 30s to protect against this.
💡 Version your cache keys
Use user:v2:42 instead of user:42. When your data schema changes, bump the version to auto-invalidate all old entries.
⏭️ Read-through isn't always better
Cache-aside gives you more control over what gets cached and when. Read-through hides complexity but makes debugging harder. Start with cache-aside.
